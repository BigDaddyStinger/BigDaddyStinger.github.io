Unity • C#

Knight Dash

Fast dash-runner with forced forward movement, obstacle dodging, and stamina management (plus bullet-hell elements).

Process: Built a tight dash loop and stamina system to push risk/reward while the knight auto-advances. Level lanes and rock obstacles create readable “micro-puzzles”; stamina gates spam-dashing, so timing matters. Playtesting focused on hitbox fairness and dash responsiveness under constant motion.

  • Tools: Unity (C#), Cinemachine, Git/GitHub, and Itch.IO.
  • Challenge: Balancing stamina drain vs. dash feel during forced movement.
  • Fix: Tuned dash i-frames & recovery, adjusted obstacle spacing, and added cooldown buffers for consistency.
  • Learned: Clear telegraphing + generous buffers beat raw difficulty for “flow” in auto-scrollers.

Unity • C# • Boss AI

Box Man Boss Fight

C#-driven state machine with 3 escalating phases, readable telegraphs, and pattern mixing.

Process:Implemented a phase-based boss using a script based state machine and health thresholds to trigger behavior swaps. Phase 1 teaches the player with simple, punishable moves. Phase 2 layers combos and an arena-wide projectile pattern. Phase 3 enrages with speed scaling and a ground-slam to break turtling. Animation Events gate hit windows. Cooldowns & anticipation frames keep it patterned.

  • Phase 1: Staple Throw (single), Kick, Normal Swing.
  • Phase 2: Adds Swing Combo 1, Swing Combo 2, 360 Swing, and a “hailstorm” multi-staple barrage.
  • Phase 3 (Enraged): Adds Slam; global boss speed ×1.25; tighter timings and shorter recovery.
  • Tools: Unity 6, C#, Animator State Machine + parameters, Animation Events, Scriptable config for phase thresholds, Cinemachine.
  • Challenge: Unintended transition loops & mistimed hitboxes during combos.
  • Fix: Explicit exit conditions per state, normalized transition durations, rooted windups, and event-driven hitbox toggles.
  • Learned: Telegraphs + recovery windows matter more than raw damage; small speed buffs (e.g., 1.25×) dramatically change pressure.

Unity • C# • Multi-Player

Fun Run

Press-to-join with Unity’s Input System, automatic player spawning, per-player cameras, and split-screen layout.

Process: Implemented PlayerInputManager for dynamic join/leave, supporting keyboard pairs and gamepads. Each spawned player gets its own PlayerInput, camera, and UI layer. A lightweight manager assigns Cinemachine brains and screen regions, and actions are enabled per-player to prevent input crosstalk.

  • Tools: Unity 6, Input System (PlayerInput/PlayerInputManager), Cinemachine, ScriptableObjects for control schemes.
  • Challenge: “Player has no camera associated” & device sharing conflicts.
  • Fix: Assign PlayerInput.camera on spawn, map devices by control scheme, and gate action maps per instance.
  • Learned: Explicit device pairing + action-map enabling eliminates ghost inputs; clean camera assignment fixes odd split layouts.
